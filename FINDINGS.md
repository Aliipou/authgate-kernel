# Security Findings — Brutal Audit 2026-05-29

Generated by 16-engineer adversarial audit. Primary sources: live attack demonstrations,
source code inspection, and runtime measurement. No summaries accepted as evidence.

## CRITICAL

### C-1: Python entity identity is a string. Zero cryptographic binding.
- File: `src/authgate/kernel/verifier.py`, `src/authgate/kernel/registry.py`
- Proof: `Entity("real-bot", MACHINE)` with same name as registered machine → PERMITTED
- Rust TCB: `subject_id = SHA-256(pubkey)` — requires private key to impersonate
- Python layer: `entity.name == registered_name` — requires knowing a string
- Impact: Every formal identity property proved in Lean4/Kani does NOT apply to Python mode

### C-2: Subprocess escape is unblocked. Proven live.
- File: `src/authgate/kernel/call_gate.py`
- Proof: Permitted tool called `subprocess.run()` and exfiltrated data
- CallGate returned `permitted=True`. Data left the process.
- Fix requires: WASM sandbox or seccomp (Linux only). Not fixed on Windows.

### C-3: Python layer has no epoch-based revocation.
- File: `src/authgate/kernel/verifier.py` — `min_epoch` does not appear
- `Action` class has no `min_epoch` parameter
- Rust TCB primary revocation: advance `min_epoch` — impossible via Python API
- Python revocation: `registry.revoke_all()` — incompatible mechanism
- Two systems with different revocation semantics both called "authgate"

### C-4: Python `verify()` is not pure. Lean4 `verify_deterministic` does not apply to it.
- File: `src/authgate/kernel/verifier.py` line 97, plus audit_log writes and tracer calls
- The theorem in `Proofs.lean` covers `engine.rs` (pure function). Python verify() has side effects.

## SEVERE

### S-1: Audit logging is optional by default. Default = zero observability.
- File: `src/authgate/kernel/verifier.py` line 97: `audit_log: object = None`
- Every example that omits `audit_log=` produces a system with no audit trail
- Fix: warn loudly or require explicit opt-out

### S-2: Audit log is unbounded in memory.
- Measured: 5.5 MB per 10k decisions = 552 MB per 1M decisions
- No rotation, no flush, no max_entries, no disk persistence in default config
- Long-running production deployments will OOM silently

### S-3: "~200 LOC TCB" claim is misleading.
- `engine.rs`: 250 LOC — correct
- Security-critical path: `engine.rs` (250) + `call_gate.rs` (412) + `dag.rs` (272) = **934 LOC**
- Total Rust crate: **7,764 LOC**
- README must say "engine.rs: 250 LOC. Full security-critical Rust path: ~934 LOC."

## MEDIUM

### M-1: Stubs return None without raising NotImplementedError.
- `MarketOracleSource.request_capability()` → always returns `None`
- `ReputationGateSource.request_capability()` → always returns `None`
- Silent failure. User gets no error. Debugs for hours.
- Fix: raise `NotImplementedError("MarketOracleSource is a stub...")`

### M-2: No JSON Schema for CanonicalAction wire format.
- External implementers cannot validate compliance without reading Rust source
- "Implementable from spec alone" (E-13 mandate) is unmet

### M-3: WASM host functions (sandbox.rs, 489 LOC) unaudited.
- Host function boundary is an attack surface equivalent to TCB
- No Kani harnesses for host function mapping

## LOW
