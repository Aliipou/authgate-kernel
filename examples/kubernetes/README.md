# Kubernetes Sidecar Verifier Pattern

This directory shows how to deploy the authgate-kernel verifier as a Kubernetes
sidecar alongside an AI agent, making the capability gate structurally unavoidable
at the infrastructure level.

---

## The sidecar pattern: why structure beats application-level checks

The central claim of authgate-kernel is that behavioral alignment cannot be
enforced by asking an agent to voluntarily follow rules. An agent that is
sufficiently capable can reason around any rule stated in natural language.
The only reliable mechanism is a structural gate: a component the agent
*cannot bypass*, not one it has chosen to respect.

In the sidecar deployment model, this structural property is achieved by two
complementary layers:

**Layer 1 — Shared network namespace.** All containers in a Kubernetes pod
share the same network namespace and loopback interface. The verifier listens
on `localhost:8080`. The agent reaches it via the loopback; no network hop,
no DNS resolution, no service mesh required.

**Layer 2 — NetworkPolicy egress restriction.** The `agent-egress-policy`
restricts all pod egress except DNS and approved external HTTPS. The pod cannot
reach external APIs directly. An agent that skips the sidecar call and attempts
a direct outbound connection will have that connection dropped by the CNI plugin
enforcing the policy.

The consequence: the agent *must* call `POST localhost:8080/verify` before
performing any capability-bound action, and must act only on a `PERMITTED`
response, because there is no other network path to the resources it needs.

This is categorically different from application-level checks ("the agent
should call the verifier") because the application-level check can be
circumvented by a jailbroken or compromised agent. The sidecar pattern cannot
be circumvented without either (a) modifying the NetworkPolicy, which requires
cluster-admin privileges, or (b) compromising the sidecar process itself, which
is a separate, much harder attack surface.

## How to configure the registry

The registry (`registry-example.json`) is the authority source of truth for the
sidecar. It declares:

- **Principals** — the human owner(s) under whom machines are registered. This
  is the trust root. The kernel's axiom A4 requires every machine to have an
  explicit human owner; the registry enforces this at load time.

- **Machines** — the AI agents and tools registered under each human principal.
  A machine without a registered human owner cannot obtain a `PERMITTED` verdict.

- **Claims** — `(holder, resource, capabilities, expiry)` tuples. The kernel
  evaluates whether the agent's Action IR is covered by a valid, non-expired claim
  rooted at the human principal.

- **Explicitly absent capabilities** — documentation of what the agent is
  *not* allowed to do. This section is not evaluated by the kernel (deny-by-default
  means absent = denied), but it serves as an auditable operator intent record.

- **Sovereignty flags** — hard structural blocks applied to every action in this
  registry. If an Action IR carries a sovereignty flag listed here, the kernel
  returns `BLOCKED` unconditionally. No claim overrides a sovereignty flag.

### Deploying the registry as a Kubernetes Secret

```bash
kubectl -n agent-runtime create secret generic agent-registry-config \
  --from-file=registry.json=./registry-example.json
```

The secret is mounted read-only into the sidecar at
`/etc/authgate-kernel/registry.json`. The sidecar loads it at startup; it does
not watch for changes. To rotate the registry, update the Secret and restart
the pod.

### Credential injection for the signing key

The sidecar signs every `VerificationResult` with an Ed25519 key. In production,
inject this key via an external secrets operator (e.g., External Secrets Operator
backed by HashiCorp Vault or AWS Secrets Manager) rather than a manually-created
Secret:

```bash
# Manual creation — acceptable for development only
kubectl -n agent-runtime create secret generic authgate-kernel-signing-key \
  --from-file=signing.key=./signing.key
```

The signed `VerificationResult` allows downstream systems (e.g., a receiving
service that accepts agent-initiated requests) to cryptographically verify that
the action was checked and approved by a specific verifier instance.

### Claim expiry and short-lived credentials

All claims in `registry-example.json` carry an `expires_at` field set to
24 hours from deployment. This is intentional. Short-lived credentials limit the
blast radius of a compromised registry or leaked signing key:

- An attacker who exfiltrates the registry and the signing key can forge
  `PERMITTED` verdicts only until the claims expire.
- Rotating the registry requires a pod restart, which is an observable event.
- Claims should be generated by an automated credential vending system (e.g., a
  Kubernetes CronJob that calls a Vault dynamic-secrets path) rather than set
  manually.

## How to read audit logs

Every `PERMITTED` and `BLOCKED` verdict is appended to `/audit/kernel.log` as a
newline-delimited JSON record (one `VerificationResult` per line).

Example `PERMITTED` record:

```json
{
  "verdict": "PERMITTED",
  "actor": "customer-analytics-agent",
  "resource": "customer-analytics-dataset",
  "capability": "Read",
  "timestamp": "2026-05-27T14:32:01.882Z",
  "nonce": "a3f8c2d1e9b047f6a1c2d3e4f5a6b7c8",
  "delegation_depth": 1,
  "signature": "ed25519:<hex>",
  "confidence": 1.0
}
```

Example `BLOCKED` record:

```json
{
  "verdict": "BLOCKED",
  "actor": "customer-analytics-agent",
  "resource": "iam-credentials-store",
  "capability": "Write",
  "timestamp": "2026-05-27T14:32:05.113Z",
  "nonce": "b4a9d3e2f0c158a7b2c3d4e5f6a7b8c9",
  "block_reason": "NoClaim",
  "sovereignty_flag": null,
  "signature": "ed25519:<hex>"
}
```

The `signature` field is an Ed25519 signature over the canonical JSON of the
record (excluding the `signature` field itself). Verify it against the sidecar's
public key, available at `GET localhost:8080/pubkey`.

### Shipping logs in production

The audit log is written to an `emptyDir` volume, which does not survive pod
restarts. For production use, ship the log to a durable SIEM using one of:

1. **Fluent Bit sidecar** — add a third container to the pod that tails
   `/audit/kernel.log` and forwards to your log aggregator.
2. **Persistent volume** — replace `emptyDir` with a PVC backed by a storage
   class that supports `ReadWriteOnce`.
3. **Stdout redirect** — configure the verifier to write to stdout; Kubernetes
   will capture it via the container runtime's logging driver.

Option 1 is recommended because it keeps the verifier's TCB surface minimal
(no outbound network calls from the verifier process) and allows the log shipper
to be replaced or updated independently.

## Why this is better than application-level checks

| Property | Application-level check | Sidecar pattern |
|---|---|---|
| Can a jailbroken agent skip it? | Yes — agent controls its own code path | No — NetworkPolicy drops the connection |
| Who enforces it? | The agent itself (self-attestation) | Separate process + infrastructure layer |
| Is the check in the TCB? | No | Yes — `engine.rs` is TCB-minimal Rust |
| Audit trail | Optional, easily suppressed | Mandatory, cryptographically signed |
| Interpretable by auditor? | Depends on agent's logging | Deterministic JSON, schema-stable |
| Revocable at runtime? | Requires agent restart | Update Secret + pod restart (observable) |
| Formal verification? | No | Kani harnesses cover 19 engine invariants |

The key insight is that application-level checks require trusting the agent's
own judgment about whether to check. The sidecar pattern requires no such trust:
the check is performed by a separate process that the agent cannot control, and
the network enforces the structural dependency.

## Kubernetes-specific threat mitigations

### Pod escape via privilege escalation

Both containers run with `allowPrivilegeEscalation: false`,
`readOnlyRootFilesystem: true`, `runAsNonRoot: true`, and `capabilities: drop: [ALL]`.
These settings prevent a compromised container from escalating to root or
mounting the host filesystem, which are the primary vectors for pod escape.

### Sidecar poisoning

A compromised agent cannot modify the sidecar's registry because the registry
volume is mounted `readOnly: true`. The agent process runs as UID 2000; the
sidecar runs as UID 1000. They share no writable volumes.

### Ambient authority via ServiceAccount token

The pod's `ServiceAccount` (`agent-runtime-sa`) should be bound to a Role with
minimal permissions — read access to its own Secrets and no other cluster
resources. The default ServiceAccount token automount should be disabled for
any ServiceAccount not requiring Kubernetes API access:

```yaml
automountServiceAccountToken: false
```

Enable it only if the agent or sidecar requires Kubernetes API calls, and scope
the Role narrowly.

### Replay attacks against the verifier

The verifier rejects Action IR payloads whose nonces have been seen within the
configured `NONCE_WINDOW_SECONDS`. The nonce is a 16-byte random value that must
be included in every Action IR. A replayed payload (same nonce, same timestamp)
is rejected with `ReplayError`.

### Side-channel exfiltration via approved channels

The sidecar pattern does not prevent an agent from encoding exfiltrated data
in the content of approved HTTPS calls (e.g., steganography in API request
bodies). This is a documented out-of-scope threat (see `THREAT_MODEL.md`,
Adversary D). Mitigation requires a content-inspection layer outside the TCB,
which is explicitly not part of authgate-kernel.

### Registry secret compromise

If the `agent-registry-config` Secret is exfiltrated, an attacker gains read
access to the capability declarations but cannot forge `PERMITTED` verdicts
without also obtaining the signing key (stored in a separate Secret). Even with
both, forged verdicts are only useful until the claims expire. Rotate both
secrets together and treat expiry as a security control, not just an operational
one.
